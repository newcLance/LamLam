## Phase 1 · AI 自动立项审核（~3 min）

> **核心原则：全自动审核，人类不参与判断，AI 独立决策准入。**

### 触发条件
收到一个游戏名（来源：心愿单 / 运营发现 / 用户提议）

### 审核流程（立项 AI 执行）

```
收到游戏名
    ↓
Step 1: 搜索确认游戏身份（英文全名、开发商、品类、区域、发售状态）
    ↓
Step 2: 查重 — 检查 data/games.json 是否已收录（含别名匹配）
    ↓  若已收录 → 终止，返回"已存在"
    ↓
Step 3: 定位官网 URL（按优先级依次尝试）
        ① 打开 Steam 商店页，从右侧边栏"访问网站"链接提取
        ② Steam 无此链接 → 搜索 "{游戏名} official website {开发商}"
        ③ 搜索结果仍无 → 判定为无独立官网，不通过
    ↓
Step 4: 硬性准入检查（自动判定，不可跳过）
    ↓  任一项 FAIL → 终止，返回"不通过 + 原因"
    ↓
Step 5: 软性内容检查（AI 读取页面后自主评估）
    ↓  综合评分 < 及格线 → 终止
    ↓
PASS → 输出立项卡片，进入 Phase 2
```

### 硬性准入标准（自动检查，一票否决）

| # | 检查项 | 规则 | 判定方式 |
|---|--------|------|---------|
| H1 | **URL 可访问性** | 官网 URL 必须能正常打开（HTTP 200，页面有实质内容） | AI 实际访问页面 |
| H2 | **非平台页** | URL 不得属于以下平台域名的详情页：`store.steampowered.com`、`store.epicgames.com`、`wegame.com.cn`、`nintendo.com/store`、`playstation.com/games`、`xbox.com/games` | 域名匹配 |
| H3 | **非空壳页** | 页面必须有实质游戏内容（不能是纯"Coming Soon"倒计时页 / 纯邮件订阅页 / 域名停放页） | AI 读取页面内容判断 |

**关于 H2 的补充说明：**
- 平台的**首页/社区/新闻**等非详情页不在禁止范围内（它们属于"平台产品"的截图范畴）
- 禁止的是用平台**商品详情页**作为游戏的官网截图——因为平台详情页模板千篇一律，不具备设计参考价值
- 如果游戏只有平台详情页而没有独立官网 → 不通过

### 软性内容检查（AI 自主评估）

立项 AI 打开官网后，读取页面内容，从以下维度自主评分：

| 维度 | 评估要点 | 权重 |
|------|---------|------|
| **视觉设计参考价值** | 页面设计是否有特色？是否值得设计师参考？有没有定制化视觉元素？ | 高 |
| **信息完整度** | 有没有游戏介绍、截图/视频、特色卖点？还是只有一张Key Art？ | 中 |
| **品牌辨识度** | 有没有独立的视觉风格/品牌调性？还是套模板？ | 中 |
| **内容更新度** | 页面内容是否有时效性？是否还在维护？ | 低 |

**综合判定：**
- AI 给出一个自然语言评估结论："通过 / 勉强通过 / 不通过"
- 勉强通过的附一句理由，交由 Lance 最终裁定
- 不通过的直接终止，附原因存档

## Phase 2 · 截图采集（~15 min）

*（与 v1.0 一致，此处简写核心规格）*

| # | 动作 | 规格 |
|---|------|------|
| 2.1 | 浏览器自动截图 | 1920px 宽，**全页长截图**，PNG 或 WebP |
| 2.2 | Cookie/弹窗自动处理 | Cookiebot/OneTrust/Didomi/年龄门禁 全覆盖 |
| 2.3 | 兜底检查 | 截图全白/全黑/被遮挡 → 自动重试 |
| 2.4 | **长图切割（必选）** | `python scripts/slice-fullpage.py --slug {slug}` |
| 2.5 | 图片标准化 | `python scripts/optimize-images.py --dir {slug}` |

### 🚨 长图切割规范（2026-04-23 起强制执行）

> **所有全页长截图必须经过 `slice-fullpage.py` 处理后才能入库。**

#### 切割规则
- **viewport 宽度 1920px** → 16:9 一屏高度 = **1080px**
- 原始长图保留为 `{prefix}-full.webp`，作为第一张图，**打 `tag: "长图"` 标签**
- 按 1080px 高度切割成 N 张切片：`{prefix}-1.webp`, `{prefix}-2.webp`, ...
- **末尾残余 ≥ 270px**（一屏的 25%）保留为独立切片
- **末尾残余 < 270px** 并入上一张（允许略超一屏）

#### 前端展示
- 第一张（长图）：独占全宽，**max-height 600px** 裁切预览 + 底部渐变提示，左上角 "全页长图" 徽章
- 后续切片：标准 2-3 列网格，每张右上角显示 `N/总数` 编号
- 点击任一图片进入灯箱查看原图

#### 脚本用法
```bash
# 处理单款游戏
python scripts/slice-fullpage.py --slug sekiro

# 处理单款游戏的特定页面
python scripts/slice-fullpage.py --slug sekiro --page home

# 处理所有游戏
python scripts/slice-fullpage.py --all

# 预览模式（不执行）
python scripts/slice-fullpage.py --slug sekiro --dry-run
```

### 截图命名规范
```
screenshots/{game-slug}/
  ├── home-full.webp                # 官网首页长图（tag: 长图）
  ├── home-1.webp ~ home-N.webp    # 官网首页 16:9 切片
  ├── about-full.webp               # 特色/玩法页长图
  ├── about-1.webp ~ about-N.webp  # 特色/玩法页切片
  ├── media-full.webp               # 媒体/预告片页长图
  └── news-full.webp                # 新闻页长图
```

### 产出物
- `screenshots/{slug}/` 目录：长图 + 切片，每个页面 5-10 张 WebP
- 更新 `data/screenshots.json`（第一张 img 为长图，tag 标记为 "长图"）

## Phase 3 · 权威优先采集（~20 min）

> **核心原则：正反馈越多越优先。阅读量、点赞量、评分权重是筛选的第一标尺。**

### 信息源分级体系

#### 第一梯队 · 权威媒体（优先采信）

| 来源 | 类型 | 权威度 | 说明 |
|------|------|--------|------|
| **IGN** | 媒体评测 | ★★★★★ | 全球最大游戏媒体，评分体系成熟 |
| **GameSpot** | 媒体评测 | ★★★★★ | 老牌权威，深度评测见长 |
| **Eurogamer** | 媒体评测 | ★★★★☆ | 欧洲视角，分析风格独特 |
| **Game Informer** | 媒体评测 | ★★★★☆ | 北美传统权威 |
| **Polygon** | 媒体评测 | ★★★★☆ | 偏文化评论视角 |
| **机核 Gcores** | 中文媒体 | ★★★★★ | 国内最专业的核心向游戏媒体 |
| **触乐** | 中文媒体 | ★★★★☆ | 严肃游戏报道 |
| **游研社** | 中文媒体 | ★★★★☆ | 深度内容+社区口碑 |

#### 第二梯队 · 高互动社区内容（正反馈筛选）

| 来源 | 筛选规则 | 说明 |
|------|---------|------|
| **Steam 评测** | 按"最有帮助"排序，优先"有用"票数 > 100 的评测 | 玩家第一手体验 |
| **Reddit** | 按 Top/Best 排序，优先 upvote > 500 的帖子或评论 | 英文社区深度讨论 |
| **B站** | 按播放量排序，优先播放量 > 10万 的视频 | 中文社区风向标 |
| **NGA** | 按回复量排序，优先精华帖和高热帖 | 国内核心玩家社区 |
| **TapTap** | 按热度/有用排序，优先高赞评价 | 国内玩家评分社区 |

#### 第三梯队 · 补充来源（仅在前两梯队不足时使用）

| 来源 | 适用场景 |
|------|---------|
| YouTube 游戏评测频道 | 头部 YouTuber 的评测视频（订阅量 > 50万） |
| X/Twitter 讨论 | 特定争议话题或开发者声明 |
| 官方 GDC/开发日志 | 美术和设计方向的一手资料 |

### 正反馈优先筛选规则

```
对于同一个事实/观点，有多个来源时：

筛选优先级 = 权威度 × 正反馈量

具体执行：
1. 权威媒体（第一梯队）的观点 > 社区内容（第二梯队）
2. 同梯队内，正反馈量高的优先：
   - 媒体：评分体系里分数更高的 / 被更多媒体引用的
   - Steam：有用票数更多的
   - Reddit：upvote 更多的
   - B站：播放量更大的
   - NGA/TapTap：点赞/回复更多的
3. 同等条件下，时效性更新的优先
4. 中文源和英文源各至少保留 1 条
```

### 采集输出要求

每条信息必须携带完整元数据：

```json
{
  "text": "观点内容",
  "src": "来源名称",
  "url": "原文链接（可直达）",
  "_tier": 1,                    // 信息源梯队：1/2/3
  "_metric": "IGN 评分 9.0",     // 正反馈指标（评分/播放量/点赞数）
  "_date": "2026-03-15"          // 信息发布日期
}
```

> **注：`_tier`、`_metric`、`_date` 为内部评审字段，不渲染到前端，仅供 Phase 5 审核时校验。**

### B站视频采集专项规则

> **🚨 关键约束：B站视频数据 100% 必须通过脚本采集，严禁 AI 自行编造 bvid/播放量。**

| 规则 | 说明 |
|------|------|
| **采集工具** | **必须使用 `python scripts/fetch-bili-videos.py --slug {slug}`**，禁止手动编写 biliVideos 数据 |
| 搜索关键词 | 脚本自动从 games.json 读取中文名 + 英文名 + 别名，各搜一次合并去重 |
| 排序方式 | 按播放量降序 |
| 筛选门槛 | 播放量 > 1万 |
| 取前 5 条 | 标题 + BV号 + 播放量 |
| **逐条验证** | 脚本自动调用 B站 view API 验证每条视频的真实性（存在性 + 标题相关性 + 时长 > 10s） |
| **内容相关性** | 必须是该游戏的实机/评测/解说内容，排除：合集混剪、无关蹭标签、纯音乐/壁纸搬运 |

#### B站采集脚本用法

```bash
# 单款游戏采集（正式写入 JSON）
python scripts/fetch-bili-videos.py --slug {slug}

# 预览模式（不写入文件，仅查看结果）
python scripts/fetch-bili-videos.py --slug {slug} --dry-run

# 批量采集所有游戏
python scripts/fetch-bili-videos.py

# 仅验证现有数据正确性
python scripts/fetch-bili-videos.py --verify-only

# 跳过指定游戏
python scripts/fetch-bili-videos.py --skip phantom-blade-zero,black-myth-wukong
```

#### 为什么必须用脚本？

2026-04-23 全面审计发现：31 款游戏中 30 款的 biliVideos 数据为 AI 幻觉编造（假 bvid 指向无关视频），
仅影之刃零（手工维护）数据正确。此次事故的根因是 Phase 3 采集阶段未强制调用 B站 API，
AI 凭记忆生成了看似合理但 100% 虚假的数据。**脚本是唯一可靠的数据来源。**

## Phase 4 · 设计师视角加工（~15 min）

> **核心原则：以游戏美术/宣发设计师为目标读者。加工目的是"提炼游戏调性，呈现玩家审美倾向"。**

### 写作身份与语境设定

你在加工内容时，假设自己是一位资深游戏视觉设计师，正在为团队整理竞品调研资料。你的读者是：
- 游戏概念设计师（需要了解竞品的视觉风格、美术方向）
- 游戏宣发设计师（需要了解竞品的品牌调性、传播卖点）
- 游戏 UI 设计师（需要了解竞品的用户界面风格、信息架构）

### 文字风格规范

| 维度 | 规则 | 正例 | 反例 |
|------|------|------|------|
| **口吻** | 设计师之间的专业交流——直接、精准、有审美判断 | "暗色调搭配高饱和霓虹，视觉冲击力极强" | "这个游戏画面很好看" |
| **术语** | 用设计/美术圈通行术语，不需要解释 | "赛博朋克 + 和风浮世绘的混搭调性" | "融合了未来科技和日本传统文化的风格" |
| **颗粒度** | 具体到能指导设计方向的程度 | "角色设计走写实比例，护甲造型参考晚明制式" | "角色设计很好" |
| **态度** | 有观点、有判断，允许主观 | "敌方阵营美术辨识度不够，boss 和杂兵视觉层级没拉开" | "敌人设计有待提升" |
| **高亮** | 核心设计关键词用 `<b>` 标注，**每段文案至少 1 个、不超过 3 个** | "主打<b>低多边形</b>+<b>手绘贴图</b>的独立美术风格" | 全文无重点 |

> **🚨 高亮硬性约束（2026-04-27 起强制执行）：**
> 
> 以下所有文本字段**必须**包含至少 1 个 `<b>` 高亮标签，否则不得进入 Phase 5：
> - `playerExp.media.*` 和 `playerExp.community.*` 的每一条 `text`
> - `story.dims[]` 的每一条 `value`
> - `aesthetic.{scene,costume,ui,symbol,promo}` 的每一条 `value`
> - `aesthetic.rivals[]` 的每一条 `reason`
> 
> 高亮目标：**核心设计概念、差异化关键词、专业术语**。加粗内容长度控制在 4-20 个字符以内，禁止整句加粗。

### 4A. 玩家体验清单（playerExp）

从"对设计师有参考价值"的角度筛选和改写原始采集数据：

| 维度 | 设计师关注点 | 条数 |
|------|-------------|------|
| `memory` 记忆点 | 什么视觉/体验让人印象深刻？可作为设计灵感的记忆锚点 | 3-5 |
| `excitement` 兴奋点 | 什么时刻让人「想截图分享」？视觉高光时刻 | 3-5 |
| `pain` 痛点 | 什么设计让人不适？UI/交互/视觉上的反面教材 | 2-4 |
| `churn` 流失点 | 什么体验让人不想继续？设计层面的致命伤 | 2-3 |

**注意：**
- 优先保留和视觉/设计/美术相关的体验观点
- 纯玩法机制的观点（如"技能平衡不好"）降低优先级，除非它影响视觉表现
- 保留原始评价的锐度，但用设计师能共鸣的方式重新表述

### 4B. IP / 故事剧情（story · 2026-05-15 升级）

独立模块，整合世界观、叙事背景和视觉基础设定。**结构上分为两层：**

```
story
├── title         一句话概括（专业向，例：功夫朋克 × 架空武侠复仇剧）
├── overview      🆕 开篇式世界观长文（小说宣传级代入感）
│   ├── tagline       一句标语，整本书的"封底金句"
│   └── paragraphs    3 段长文，依次回答三个核心问题
└── dims          7 维度参数化详情（保持原有专业向数据）
```

#### 4B-1. overview · 开篇式世界观长文（必填）

> **核心原则（v2.3 升级）：宣发文案不是内容摘要（Summary），而是商业广告（Advertisement）。**
> **它的唯一目的：在 3 秒内抓住眼球，在 10 秒内建立期待，在 30 秒内促成转化（点击/购买/想立刻去玩这款游戏）。**
>
> 写作目的对标"<b>小说发售前的内封简介 / 院线电影前导预告片旁白 / Steam 商品页置顶宣发文</b>"——
> 不是冷冰冰的设定罗列，而是用**信息密度+情绪冲击+悬念钩子**把读者瞬间拽进世界观。
>
> **🚨 v2.3 重要变更：`story.title` 已废弃。** tagline 已承担"封底金句 + 高概念碰撞"双重职责，title 与 tagline 高度重复时必须删除 title，避免冗余信息干扰广告级转化。新接入游戏不再写 title 字段。

##### 内容采集三问题（Phase 3 阶段就要带着这三个问题去筛信源）

| # | 问题 | 写到第几段 |
|---|------|-----------|
| Q1 | **这是个什么样的世界？** —— 时代、文化底色、整体氛围、能让人一下子想象出画面的核心意象 | 第 1 段（世界图景） |
| Q2 | **主要角色有哪些？** —— 主角是谁、处境如何、还有哪几股关键势力 | 第 2 段（主角处境） |
| Q3 | **角色之间有没有推动剧情发展，是如何推动的？** —— 阵营对抗、人物关系冲突、戏剧张力的来源 | 第 3 段（角色冲突 + 剧情驱动） |

##### 字段结构

```json
"story": {
  "overview": {
    "tagline": "六十六天，一柄残刀，一个被江湖弃绝的名字。",
    "paragraphs": [
      "第 1 段：世界图景（120-180 字）",
      "第 2 段：主角处境（120-180 字）",
      "第 3 段：角色冲突与剧情驱动（150-220 字）"
    ]
  },
  "dims": [ ... ]
}
```

> v2.3 起 `story.title` 字段已废弃。如果是历史游戏数据中遗留的 title 与新版 tagline 重复，必须删除 title。

##### 🎯 五维钩子（Hook）法则（v2.3 强制）

> **写完 overview 必须逐条对照五个原则自检。任何一条不达标 → 退回重写。**
> 这五条不是建议，而是宣发文案的"五条硬骨架"，缺一不可。

###### 原则一：高概念碰撞（The High Concept）—— 视觉与认知的奇观

**要求：** 用一句话说明这款游戏"凭什么是不可替代的"。
现代读者面临严重的信息过载，传统题材标签（武林复仇 / 末世废土 / 星际战争）已经无法刺激神经。
你必须把**两个看似无关甚至相悖**的元素砸在一起，让读者脑海中瞬间弹出电影画面。

**执行公式：**【熟悉的基底】+【异常的变量】

| 反面教材 | 正面示范 |
|---------|---------|
| 这是一个关于古代刺客复仇的故事，他武功高强，门派林立 | <b>功夫朋克</b> × 架空武侠 —— 古风水墨 + 蒸汽机械义肢 |
| 一个赛博朋克世界里的私家侦探故事 | <b>赛博佛教</b> × 灵魂买卖 —— 霓虹寺庙里的转世交易所 |

**自检问题：** 你的文案里，有没有一个能让读者脑海中**瞬间弹出电影画面**的"奇观设定"？
**落点：** 通常出现在第 1 段的开头或 tagline 中，作为整篇文案的"视觉锚点"。

---

###### 原则二：极限处境（The Extreme Stakes）—— 倒计时或绝境

**要求：** 主角必须**一出场就踩在悬崖边上**，没有退路。
平铺直叙的开局是宣发的死敌。读者不关心主角有多强，**读者只关心主角有多惨、面临多大的危机**。
有了极限处境，剧情张力才出得来。

**执行标准：** 设定明确的代价 / 倒计时 / 无法调和的两难选择。

| 反面教材 | 正面示范 |
|---------|---------|
| 主角踏上了寻找仇人的漫长旅途，面临重重考验 | 伤重将死，体内只剩<b>六十六天气数</b>；全世界都在追杀他 |
| 主角即将面对最强大的敌人 | 救女儿要 24 小时穿过感染区，途中每停一秒她体温就降 0.1 度 |

**自检问题：** 读者能感觉到主角身上的**"紧迫感"**吗？如果主角失败了，**代价够不够惨痛**？
**落点：** 第 2 段必须出现至少 1 个具体的代价/倒计时/绝境数字。

---

###### 原则三：灰度悖论（The Character Nuance）—— 打破扁平人设

**要求：** 不要介绍好人多好、坏人多坏，**要写出他们的复杂性**。
当代读者更偏爱全员恶人 / 悲情反派 / 有缺陷的英雄。
文案中必须给核心配角或反派一两笔群像刻画，**展现立场的撕裂**。

**执行标准：** 为人物贴上"反差标签"。

| 反面教材 | 正面示范 |
|---------|---------|
| 邪恶的怪面组织妄图统治武林，主角的朋友们奋力抵抗 | <b>替你挡刀的旧友</b>，也会举刀向你；怪面首领甘愿变成怪物 |
| 反派是一个想毁灭世界的疯子 | 他烧掉的每一座城，都是为了找回死去那年她的样子 |

**自检问题：** 你的反派**有让人同情的理由**吗？主角的盟友**有背叛的可能**吗？
**落点：** 第 3 段必须出现至少 2 个角色/势力，且至少 1 个带"反差描述"。

---

###### 原则四：情绪内核（The Emotional Core）—— 超越故事本身的"痛点"

**要求：** 故事是壳，情绪才是让读者掏钱的灵魂。
这是宣发文案的"**定海神针**"。你必须用一句话告诉读者：
除了打怪升级谈恋爱，这款游戏到底在探讨什么人类共同的情感？是孤独？宿命？求而不得？

**执行标准：** 使用「**这不是一个关于 A 的故事，而是一个关于 B 的故事**」句式（或等价句式）。
其中 A 是表层题材，B 是深层情绪母题。

| 反面教材 | 正面示范 |
|---------|---------|
| 这是一个精彩绝伦的复仇传奇，展现了武侠的魅力 | 这不是一个关于复仇的故事，是一个关于「<b>来不及</b>」的故事 |
| 这是一个史诗冒险，主角将拯救世界 | 这不是一个关于英雄的故事，是一个关于「**没人记得我做过这些**」的故事 |

**自检问题：** 读完最后一句，读者心里**会产生一种隐隐作痛或热血沸腾的"余味"**吗？
**落点：** 第 3 段末尾，作为整篇文案的"情绪定海神针"。**A/B 必须形成强反差**——A 是大众预期，B 是反预期的深层议题。

---

###### 原则五：信息留白（The Information Gap）—— 用疑问代替陈述

**要求：** **永远不要在文案里剧透结局**，要把读者推向迷宫的入口。
文案的结尾**绝不能是一个句号**，必须是一个问号 / 戛然而止的悬念 / 隐含致命选择的暗示。
要制造一种"**不点进去看就浑身难受**"的阅读饥渴感。

**执行标准：** 抛出致命问题 / 暗示惨烈走向 / 留下开放式选择。

| 反面教材 | 正面示范 |
|---------|---------|
| 最终，他查明了真相，打败了幕后黑手，拯救了江湖 | 那条若隐若现的<b>红色丝线</b>，缠住了天下侠客的咽喉。你怎么打、怎么选、放过谁、错信谁？ |
| 主角最终找到了答案 | 她在最后一秒认出了那个人——但你不会，<b>除非你亲自走到那一步</b> |

**自检问题：** 文案的最后一句话，是否构成了**强烈的"行动召唤（Call to Action）"或"解谜冲动"**？
**落点：** 第 3 段末尾必须以问号/疑问句/省略号/开放式选择收束。禁止以句号 + 陈述总结收尾。

---

写作完成后，必须能在 overview 中**逐条圈出**对应钩子的落点：

|---|------|---------|--------------------|
| ① | 高概念碰撞 | 第 1 段 / tagline | "________" |
| ② | 极限处境 | 第 2 段 | "________" |
| ③ | 灰度悖论 | 第 3 段中段 | "________" |
| ④ | 情绪内核 | 第 3 段末尾 | "________" |
| ⑤ | 信息留白 | 第 3 段末尾 | "________" |

**任一钩子无法明确指出落点 → Phase 5.9 自动审核必然 FAIL → 退回 4B-1 重写。**

##### 禁用清单（出现即退回重写）

- ❌ "本作"、"该作"、"该游戏"、"玩家将"
- ❌ "在...的世界中"、"在...的设定下"、"以...为背景"
- ❌ "讲述了..."、"围绕...展开"、"以...为核心"
- ❌ "精彩绝伦"、"史诗般的"、"传奇巨作"、"经典之作" 等空洞褒词
- ❌ 任何形式的玩法机制说明（"战斗系统采用...""核心循环是..."），玩法归 `worldview.combat / coreLoop` 写
- ❌ 任何形式的"评分"、"销量"、"获得 X 奖"等市场数据
- ❌ 任何剧透结局、揭示反转、点明真相的内容（违反原则五）

##### 长度硬约束

- `tagline`：12-30 字，独立一句话，能单独印在海报上，**必须包含原则一或原则二的钩子**
- `paragraphs`：3 段，单段 120-220 字，**总字数 380-600 字**
- 段间必须分段（不允许 3 段揉成 1 段），每段聚焦一个核心问题（Q1/Q2/Q3）

#### 4B-2. dims · 参数化详情（v2.3 刷新 · 8 维度）

> **核心原则：每条聚焦一个维度，避免与 overview 重复；颗粒度到能直接指导美术执行的程度。**

| 维度 | 要回答的问题 | 写法要求 |
|------|-------------|---------|
| 时间 | 故事发生在什么年代/时间线？ | 精确到对美术风格有影响的时代标签（如"明末清初中后期叠加蒸汽工业初萌"） |
| 发生地 | 故事发生在哪里？ | 地理/文化背景 + **色彩基调**，直接关联场景美术方向 |
| 角色扮演 | 主角是谁？造型走什么路线？ | 侧重**外观设计风格**（服装版型/材质/气质），而非剧情身份 |
| 科技水平 | 世界观科技分类？是全民还是局部？ | 标签式 + **叠加层级**说明（如"局部科技越级，非全民赛博朋克"） |
| 对抗关系 | 阵营之间的**材质语言**差异？ | 对比式，三方阵营各用一组关键材质词区分（如"布衣皮革 vs 金属外骨骼 vs 神秘符号"） |
| 地图类型 | 地貌分类 + 关卡共性 | 标签列举 + 抽象设计共性（如"垂直立体地形是核心、幽闭与高差是关卡共性"） |
| 视觉母题 | 反复出现的核心视觉锚点？ | 列出 3-5 个反复出现的图像符号 + 色彩母题（如"红色丝线 / 面具 / 齿轮咬合"+"水墨黑灰 × 锈蚀金红"） |
| 美学谱系 | 灵感来源谱系？自我定位标签？ | 列出 3-5 部作品参考 + 自我美学标签（如"《铳梦》+《剑风传奇》+《攻壳机动队》→ 自我定位「功夫朋克」"） |

> **v2.3 关键改动：**
> - "元素" 改名 "视觉母题"（语义更准）
> - 新增 "美学谱系"（灵感来源从原"元素"中独立出来，便于设计师快速建立参考库）
> - 字数控制每条 70-110 字，加粗 ≤ 30 字

#### 4B-3. 与 worldview 的边界（重要）

| 模块 | 视觉位置 | 信息颗粒度 | 阅读目的 |
|------|---------|-----------|---------|
| `worldview` | 时间线下方 | 4 段通俗速写 | 让设计师 90 秒知道"这游戏长什么样" |
| `story.overview` | IP/故事剧情顶部 | 3 段开篇宣传文 | 让设计师 30 秒被拉进世界观，产生情绪共鸣 |
| `story.dims` | story.overview 下方 | 7 维度专业参数 | 设计师后续查具体设定细节 |

**写作顺序建议：dims（先把骨架捋清楚） → overview（再用文学手法包装） → worldview（最后用大白话翻译）**

### 4C. 玩家审美偏好（aesthetic · v2.4 升级）

> **🚨 核心审核口径（v2.4 新增）：**
> "在版号常态化、买量成本飙升的时代，美术风格如果不能在 **刷抖音的 0.5 秒内** 让玩家停下手指，
> 这个游戏在宣发层面就已经输了。"
>
> 玩家审美偏好模块是为了帮助设计师**总结视觉美学定调原则**。
> 采集 / 分析 / 审核必须严格围绕"**0.5 秒抓眼球**"这个核心目标展开。

#### 4C-0. 五大美术原则（强制标准）

> **🎯 每款游戏的 aesthetic 都必须明确命中以下五大原则。**
> 写完后必须能在 5 维度卡片中**逐条圈出对应原则的落点**，缺一不可。

##### 原则一【80/20 法则】—— 熟悉的基底 + 致命的变异

| 项 | 内容 |
|----|------|
| **目的** | 降低认知门槛的同时，制造极强新鲜感 |
| **标准** | 美术大基调必须 **80% 建立在玩家极度熟悉的文化锚点**（中式水墨、九龙城寨、维多利亚...），用 **20% 极具破坏感、反常识的变异元素**去撕裂它 |
| **公式** | 【熟悉的基底 80%】+【致命的变异 20%】= 一句"A × B"的视觉奇观 |
| **正例** | 影之刃零 = 80% 斗笠/古塔/刀剑（武侠） + 20% 齿轮/机械义肢/排气管（蒸汽朋克） |
| **自检** | 我们的美术设定里，有没有一个能用"A × B"公式一句话讲清楚的视觉奇观？ |
| **主要落点** | `scene` 场景氛围 + `symbol` 视觉符号 |

##### 原则二【色彩刺客】—— 绝对的色彩主权与极简调色盘

| 项 | 内容 |
|----|------|
| **目的** | 在任何平台（包括被压缩到糊的短视频画质下），玩家一眼认出这是哪款游戏 |
| **标准** | 确立 1 个 **"环境主色调"**（低明度灰黑/暗青...）+ 1 个 **"高亮情绪色"**（高饱和血红/荧光黄...）；**其他色彩必须被极度压抑** |
| **公式** | 【环境主色调】+【高亮情绪色】= 双色锁定的视觉记忆点 |
| **正例** | 《P5》= 红黑；《只狼》= 枯黄 + 暗灰；影之刃零 = 暗黑底色 + 高对比红光 |
| **自检** | 把游戏截图缩到**邮票大小** / 加**高斯模糊**，玩家还能认出这是我们的游戏吗？ |
| **主要落点** | `scene` 场景氛围（必须在 value 里写出"主色调 + 情绪色"双色组合） |

##### 原则三【剪影定生死】—— 极致的材质碰撞与轮廓反差

| 项 | 内容 |
|----|------|
| **目的** | 无需看清脸部细节，仅凭轮廓和材质质感就能传递阵营/性格/战斗方式 |
| **标准** | 主角与敌人的设计在 **剪影形状** 和 **表面材质** 上形成戏剧性对立 |
| **公式** | 【主角剪影 × 主角材质】vs【敌方剪影 × 敌方材质】= 涂黑后仍能秒辨 |
| **正例** | 影之刃零 = 主角"流线型/收敛/布料破败" vs Boss"张狂/不对称/工业外骨骼" |
| **自检** | 把所有角色涂成**全黑剪影**，玩家还能分清谁是主角、谁是杂兵、谁是 Boss 吗？ |
| **主要落点** | `costume` 角色服装（必须用对比式描述，而不是单边夸赞） |

##### 原则四【信息磨损】—— 拒绝无菌感，用"战损"做视觉叙事

| 项 | 内容 |
|----|------|
| **目的** | 让画面自带故事感，引发玩家好奇心与探索欲 |
| **标准** | 任何出现在核心视觉中的物件，必须带有**被时间或暴力摧残过的痕迹**——武器有缺口血污、建筑有风化修补、机械有锈蚀漏油 |
| **公式** | 【新洁的塑料感】→ 退回；【风霜战损 + 修补痕迹】→ PASS |
| **正例** | 影之刃零 = 残刀缺口 + 锈蚀管道 + 衣襟焦痕 |
| **自检** | 画面细节里，是否藏着这个世界**过去发生过灾难/战斗的证据**？ |
| **主要落点** | `scene` 场景氛围 + `costume` 角色服装 + `symbol` 视觉符号（任意 ≥ 2 项必须出现"磨损/战损/锈蚀/残破"类描述） |

##### 原则五【UI 即世界】—— 沉浸式留白，UI 绝不抢夺美术焦点

| 项 | 内容 |
|----|------|
| **目的** | 提升高级感，把玩家注意力 100% 集中在核心美术表现上 |
| **标准** | 摒弃多余 HUD；UI **极简、扁平**，甚至**融入世界观**（Diegetic UI，例如生命值通过衣服破损或背后发光管显示） |
| **公式** | 【满屏 HUD/巨大伤害数字/花哨头像框】→ FAIL；【极简 + 世界观融入】→ PASS |
| **正例** | 影之刃零 = 极简战斗 HUD + 墨迹质感 + 用喷血量/受击硬直反馈打击感 |
| **自检** | UI 是不是在努力**"隐藏自己"**？是否符合整体色彩基调，没有破坏沉浸感？ |
| **主要落点** | `ui` UI 界面（必须明确写出"是否极简 / 是否世界观融入 / 信息密度多大"） |

---

#### 4C-1. summary · 美学综述（v2.5 新增 · 必填）

> **核心原则：在所有具体卡片之上，先用一段宏观综述告诉设计师"这游戏属于什么美学流派、它从哪儿来、可以借鉴谁"。**
> 综述区是整个 aesthetic 模块的**统领段**——读者读完它，应该能在脑子里建立这款游戏的美学坐标系。

##### 内容采集三问题（Phase 3 阶段就要带着这三个问题去筛信源）

| # | 问题 | 落到哪个字段 |
|---|------|------------|
| Q1 | **这是什么美学？** —— 给它一个能海报级使用的标签词，再展开讲清调性 | `label` + `definition` |
| Q2 | **审美的共识是怎么发展出来的？** —— 这套美学不是凭空冒出来的，要追溯它的视觉文化谱系 | `evolution` |
| Q3 | **过去有哪些类似的游戏 / 电影 / 电视剧 / 动漫值得美术借鉴？** —— 给设计师一个直接可调用的参考库 | `references[]` |

##### 字段结构

```json
"aesthetic": {
  "summary": {
    "label": "功夫朋克 Kungfu Punk",
    "definition": "Q1 答案：120-180 字定义这套美学的调性和视觉特征",
    "evolution": "Q2 答案：150-280 字回顾审美共识的发展脉络",
    "references": [
      { "type": "manga|anime|film|game|show", "title": "作品名", "year": "1990", "note": "可借鉴点（带 <b>高亮</b>）" }
    ]
  },
  "scene": { ... },
  ...
}
```

##### 三问写作规范

###### Q1 · `label` + `definition`

| 字段 | 要求 |
|------|------|
| `label` | 6-20 字。**给这套美学一个能印在海报上的标签词**——可以是已有概念（赛博朋克、维多利亚哥特），也可以是原创复合词（功夫朋克、霓虹武侠）；中文 + 英文双标更佳 |
| `definition` | 120-180 字。回答"这是什么美学"——必须包含：① 主基底（80% 熟悉的文化锚点）② 变异元素（20% 反常识的新东西）③ 双色 DNA（呼应原则②）④ 整体调性形容（介于 X 与 Y 之间） |

###### Q2 · `evolution`

| 字段 | 要求 |
|------|------|
| `evolution` | 150-280 字。回答"审美共识如何发展出来"——必须包含至少 **2 个明确的时间节点 / 阶段**（如"1980-90 年代港片 → 2000s 日漫赛博 → 2010s 国际化暗黑动作"），并且引用至少 **3 部具体作品**作为节点锚定。**禁止空泛说"自古以来" / "由来已久"**。 |

###### Q3 · `references[]`

| 字段 | 要求 |
|------|------|
| `type` | 必须是以下之一：`game` 游戏 / `film` 电影 / `anime` 动画 / `manga` 漫画 / `show` 剧集 |
| `title` | 作品名（中英可混排，主推中文） |
| `year` | 首发年份（漫画/系列电影可用区间，如 "2000 / 2013"） |
| `note` | **借鉴点**——20-50 字，必须聚焦"美术设计师能从这部作品里偷什么"，而不是剧情概述；至少 1 个 `<b>` 高亮 |

##### 数量与多样性硬约束

- `references` **数量 5-8 条**
- 类型多样性：必须**至少覆盖 3 种 type**（不能全是 game，也不能全是 film）
- 时间跨度：**最早作品和最晚作品的年份差 ≥ 20 年**（保证审美谱系纵深）
- 每条 `note` 必须给出"借鉴点"，禁止只写作品介绍

##### 禁用清单

- ❌ "由来已久" / "自古以来" / "源远流长"
- ❌ "致敬经典" / "向...致敬"（空话，不写"致敬什么"）
- ❌ 只列作品名而不写借鉴点的 references 条目
- ❌ definition 出现"独一无二""绝无仅有""前所未有"等空洞自夸

#### 4C-2. 五维卡片字段结构（落地承载层）

**5 个维度，每个维度一张卡片，各含 `title`（一句话概括）+ `value`（展开描述）+ 🆕 `anchorRule`（命中的原则编号数组）：**

| 维度 key | 定位 | 写法要求 | 必须命中的原则 |
|----------|------|---------|---------------|
| `scene` 场景氛围 | 视觉环境整体调性 | 色调、光影、空间感、地貌、天气/时间；**必须包含"主色调 + 情绪色"双色组合** | 原则①、②、④ |
| `costume` 角色服装 | 角色外观核心风格 | 材质、配色、文化参考、阵营差异；**必须用"主角剪影/材质 vs 敌方剪影/材质"对比句式** | 原则①、③、④ |
| `ui` UI 界面 | 界面设计语言 | HUD 风格、菜单质感、字体、信息密度；**必须明确是否极简 / 是否 Diegetic** | 原则⑤ |
| `symbol` 视觉符号 | 反复出现的核心视觉锚点 | Logo、图腾、色彩母题、材质符号、叙事性视觉元素 | 原则①、② |
| `promo` 宣发风格 | 官方宣传物料设计风格 | 预告片剪辑节奏、Key Art 构图、品牌色、文案调性 | 原则①、② |

##### 字段结构

```json
"aesthetic": {
  "scene": {
    "title": "暗黑东方水墨 × 蒸汽工业废墟",
    "value": "...（要素覆盖：双色锁定 + 战损叙事 + 80/20 公式）",
    "anchorRule": [1, 2, 4]
  },
  "costume": {
    "title": "布衣侠客 vs 蒸汽机械怪面",
    "value": "...（要素覆盖：主角 vs 敌方剪影/材质对比）",
    "anchorRule": [1, 3, 4]
  },
  "ui": { "..." : "...", "anchorRule": [5] },
  "symbol": { "...": "...", "anchorRule": [1, 2] },
  "promo": { "...": "...", "anchorRule": [1, 2] },
  "rivals": [...]
}
```

> **🚨 `anchorRule` 是内部审核自检字段，不渲染到前端。** 它只用于 Phase 5.11 自动校验"声明的原则是否在 value 文本中实际命中"。前端展示时只显示标题 + 段落，让设计师读到的是自然文案，而不是审核标签。

##### 写作硬约束

- 每条 `value` 字数 **80-180 字**
- 每条 `value` 至少 **2 个 `<b>` 高亮**
- 每条 `value` 必须可以**逐字段对应到 `anchorRule` 中声明的原则**——审核时会逐条核对，对不上即 FAIL
- 加粗内容长度 ≤ 30 字
- 禁用空洞褒词："唯美""精致""华丽""极致""绝美""惊艳"

### 4D. 竞品推荐（aesthetic.rivals）

> **核心原则：竞品必须从三个已采集模块中提取线索，且所有出现的竞品都必须在站内有对应详情页。**

#### 选取规则（按优先级排序）

**① 从"玩家游戏体验清单"选取**

| 线索来源 | 选取逻辑 | 示例 |
|----------|---------|------|
| 记忆点/兴奋点中的类比 | 媒体或玩家说"XX 的战斗堪比《鬼泣5》"，那《鬼泣5》就是候选竞品 | "战斗满足感已可比肩《鬼泣5》" → DMC5 |
| 流失点中的对标 | 玩家说"相比《只狼》，这款的弹反反馈差远了"，那《只狼》就是候选竞品 | "不如《只狼》的弹反手感" → Sekiro |
| 痛点中的参照 | 玩家拿其他游戏作为正面/反面参照时，被提及的游戏都是候选 | |

**② 从"玩家审美偏好"选取**

| 匹配维度 | 选取逻辑 | 示例 |
|----------|---------|------|
| 相似美术风格 | 场景氛围 / 角色服装 / UI 设计语言接近 | 同为暗黑东方水墨调 |
| 相似文化背景 | 同一文化圈 / 同一时代审美 | 同为中国武侠题材 |
| 共同美学源头 | 受相同作品/流派影响 | 同受《铳梦》《剑风传奇》影响 |

**③ 从"IP / 故事剧情"选取**

| 匹配维度 | 选取逻辑 | 示例 |
|----------|---------|------|
| 相似世界观 | 科技水平/时代/文明类型接近 | 同为"古代 + 蒸汽朋克"混合层 |
| 相似对抗关系 | 阵营结构和视觉对比手法相近 | 同为"人类 vs 变异/改造者"双方阵营 |
| 相似叙事框架 | 主角处境/故事驱动力相似 | 同为"复仇驱动 + 倒计时/时限"叙事 |

#### 选取流程

```
Step 1: 从 playerExp 提取所有被提及的游戏名
        （扫描 memory/excitement/pain/churn 中的文本引用）
    ↓
Step 2: 从 aesthetic 5 维度和 story 维度中，
        在当前数据库的游戏中匹配"最接近"的候选
    ↓
Step 3: 合并候选池，按以下权重排序：
        - 被 playerExp 直接提及（权重最高）
        - 审美偏好多维度匹配（权重次之）
        - 世界观/故事匹配（权重第三）
    ↓
Step 4: 取 TOP 4，每款附理由
        （理由必须注明来自哪个维度的匹配）
    ↓
Step 5: 检查 4 款竞品是否都在 data/games.json 中
        ├── 在 → 直接引用 id
        └── 不在 → 触发"缺失竞品接入流程"
```

#### 理由写法规范

每条理由必须标明匹配来源（不能笼统说"整体类似"）：

```json
{
  "id": 153,
  "reason": "媒体评测多次对标其<b>弹反系统</b>，审美上同为<b>暗黑东方美学</b>，世界观同属架空古代+超自然力量",
  "_matchDims": ["playerExp.media.excitement", "aesthetic.scene", "story"]
}
```

> `_matchDims` 为内部审核字段，不渲染到前端，供 Phase 5 验真时校验竞品选取有据可查。

#### 缺失竞品自动接入流程

当选出的 4 款竞品中有不在当前数据库中的游戏时：

```
发现竞品 "Game X" 不在 data/games.json
    ↓
自动触发 Phase 1 立项审核（AI 自动，不需人工）
    ├── 通过 → 走完整 Phase 1-6 接入流程
    │          （优先级高于其他待接入游戏）
    └── 不通过 → 从候选池中选取下一个替补竞品
                 （不通过的原因通常是：无独立官网）
    ↓
接入完成后，回填 rival.id 到原游戏的 JSON 中
    ↓
重新构建站点，确保竞品链接可跳转
```

**关键约束：发布时，所有 rivals 中的 id 必须在 data/games.json 中有对应条目，零死链。**

### 4E. 游戏世界观（worldview · 2026-05-15 新增）

> **核心原则：四维度速写——让设计师 90 秒读完就知道"这游戏长什么样"。**
> 
> **此模块在前端展示位置：时间线 ↓ 体验清单之间。**

#### 维度定义

| key | 维度名 | 要回答的问题 | 写作侧重 |
|-----|--------|-------------|---------|
| `background` | 游戏背景 | 故事发生在哪里？什么世界？什么调性？ | 世界观底色、文化混搭、整体氛围 |
| `mainStory` | 主线剧情 | 玩家在干嘛？驱动力是什么？目标是什么？ | 主角处境、戏剧张力、叙事钩子 |
| `combat` | 战斗系统 | 打架是什么感觉？爽点和门槛在哪？ | 核心动作机制、手感反馈、上手曲线 |
| `coreLoop` | 核心循环 | 一个小时玩起来在循环做什么？ | 推进节奏、奖励反馈、玩家压力源 |

#### 字段结构

```json
"worldview": {
  "background": {
    "title": "一句话专业概括",
    "value": "通俗易懂的展开版（必须口语化，禁止书面化）",
    "keywords": ["3-5 个", "核心标签"]
  },
  "mainStory": { ... },
  "combat": { ... },
  "coreLoop": { ... }
}
```

#### 写作规范（与 story / aesthetic 显著不同）

| 维度 | story / aesthetic 已有规则 | worldview 的额外要求 |
|------|--------------------------|---------------------|
| 口吻 | 设计师之间专业交流 | **`title` 保留专业感，`value` 必须像跟朋友讲游戏** |
| 颗粒度 | 具体、能指导设计 | `value` 段落式叙述，不要切碎成短词标签 |
| 长度 | 短句紧凑 | `value` 建议 120-220 字，**完整成段** |
| 高亮 | 每条 ≥ 1 个 `<b>` | 每个 `value` ≥ 2 个 `<b>`（信息密度更高） |

#### 通俗化硬规则（最终审核重点）

> **🚨 v2.1 起强制：worldview 四维度的 `value` 字段必须通过"通俗化审核"，否则不得发布。**

每条 `value` 必须满足：

1. **口语化句式**——可以用"你扮演...""玩起来就是..."，**禁止**"该作以...为核心""通过...的方式呈现"
2. **零行业黑话**——能用"打架"就不要写"战斗反馈机制"，能用"压力"就不要写"压迫感叙事张力"
3. **场景化举例**——至少出现一处"比如...""下一秒..."等具体画面，而不是抽象概念堆叠
4. **避免书面化连接词**——禁用"此外、然而、综上所述、值得一提的是、不仅...而且..."等文章式表达
5. **接得住"这游戏到底什么样"**——拿给一个没玩过的朋友读，他能复述出大概在干嘛

##### 反例 → 正例对照

| ❌ 书面化反例 | ✅ 通俗版正例 |
|-------------|------------|
| "本作以高难度动作战斗为核心玩法机制，强调玩家对敌人攻击模式的精准识别与反应。" | "战斗就是<b>看招拆招</b>：盯住敌人出招，按对时机弹反，节奏对了一套连招打出去特别爽。" |
| "故事背景设定于一个融合了东方武侠美学与蒸汽朋克元素的架空世界。" | "故事发生在一个<b>架空的古代中国</b>——但这里的江湖跟你想的不太一样，一群高手把自己改造成了半人半机械的怪物。" |
| "核心循环以关卡推进、Boss 战胜利、装备解锁的螺旋上升模型展开。" | "玩起来就是一个<b>关卡制循环</b>：进区域 → 推剧情 → 干掉 Boss → 拿新武器 → 进下一关。" |

#### 采集与生产顺序

```
Phase 3 采集时，story 和 worldview 共用同一批信息源
    ↓
Phase 4B 先写 story（设计师专业向 · 表格式）
    ↓
Phase 4E 基于 story + 媒体试玩 / 玩家社区中的"玩起来感受"提炼 worldview
    ↓
Phase 5 双重校验：高亮覆盖率 + 通俗化审核
```

#### 与 story 模块的边界

- **story**：表格式、专业向、维度更细（时间/发生地/角色/科技/对抗/地图/元素），是"<b>世界设定档案</b>"
- **worldview**：段落式、通俗向、四维度（背景/主线/战斗/循环），是"<b>给朋友的口头介绍</b>"
- **两者数据不重复，互为补充**：story 给设计师查参数，worldview 给所有人快速建立印象

## Phase 5 · 数据兜底验真（~15 min）

> **核心原则：逐条验证数据正确性。链接必须可达，内容必须对版，容错为零。**

### 审核执行者
- **一级审核（自动）**：AI 逐条机器验证，生成验真报告
- **二级审核（人工）**：Lance 审阅报告 + 抽查关键项

### AI 自动验真清单

#### 5.1 链接可达性验证

```

  ├── HTTP 请求验证：返回 200 → PASS
  ├── 返回 301/302 → 跟随重定向后再验
  ├── 返回 403/404/5xx → FAIL，标记为"链接失效"
  └── 超时 (>10s) → FAIL，标记为"链接超时"

输出：
  ✓ 25/27 links valid
  ✗ 2 links failed:
    - community.pain[2].url → 404
    - media.memory[0].url → timeout
```

```
  ├── 检查页面内容是否与引用的 text 语义匹配
  │   （允许改写/提炼，但核心事实不能偏离）
  ├── 检查 src（来源名称）是否与实际域名/品牌匹配
```


#### 5.5 高亮覆盖率验证（2026-04-27 新增 · 必检项）

> **🚨 此步骤为自动化硬卡点，FAIL 则不允许进入 Phase 6。**

```
对 data/games/{slug}.json 中以下字段逐条检查 <b> 标签：

  ├── playerExp.media.{memory,excitement,pain,churn}[*].text
  │   └── 每条必须包含至少 1 个 <b>...</b> → 无则 FAIL
  ├── playerExp.community.{memory,excitement,pain,churn}[*].text
  │   └── 每条必须包含至少 1 个 <b>...</b> → 无则 FAIL
  ├── story.dims[*].value
  │   └── 每条必须包含至少 1 个 <b>...</b> → 无则 FAIL
  ├── story.overview.paragraphs[*]
  │   └── 全 overview 累计 <b>...</b> ≥ 6 个 → 不足则 FAIL
```

##### 5.9.2 五维钩子逐项校验（自动）

```

  ├── ① 高概念碰撞（The High Concept）

  │         例：功夫 × 朋克、武侠 × 蒸汽、佛教 × 赛博

  │     - 概念过于常见且无新角度（"末世废土"/"修仙世界"裸用）
  │
  ├── ② 极限处境（The Extreme Stakes）
  │   判定：第 2 段必须出现至少 1 个"具体代价 / 倒计时 / 两难选择"
  │         具体代价 = 数字化死亡威胁/失去物（"66 天"/"24 小时"/"剩 3 颗子弹"）
  │         倒计时 = 明确的时间约束

  │     - 主角处境不紧迫（"踏上漫长旅途"）
  │
  ├── ③ 灰度悖论（The Character Nuance）
  │   判定：第 3 段必须出现至少 2 个角色/势力，且至少 1 处"反差描述"
  │         反差描述 = 同一对象的两种相反属性并置
  │         例："替你挡刀的旧友，也会举刀向你"

  │     - 反派被描述为纯粹"邪恶/疯狂/想毁灭世界"

  │     - 仅出现 1 个角色，缺少势力对峙
  │
  ├── ④ 情绪内核（The Emotional Core）
  │   判定：第 3 段末尾必须出现"A vs B 反差句式"或等价的情绪母题点睛
  │         标准句式：「这不是一个关于 A 的故事，是一个关于 B 的故事」
  │         等价句式：可以变形（"这从来不是 X，而是 Y"），但 A 必须是表层题材，
  │                   B 必须是深层情绪母题（孤独/宿命/求而不得/来不及/被遗忘等）
  │   FAIL 触发：
  │     - 缺失情绪点睛句
  │     - A = B 没有形成反差（"这不是复仇故事，是关于复仇的故事"）
  │     - B 是另一个题材标签而非情绪母题（"这不是冒险，是战斗"）
  │
  └── ⑤ 信息留白（The Information Gap）
      判定：第 3 段最后一句必须满足以下任一条件：
            - 以问号结尾（疑问句/反问句）
            - 以省略号结尾（戛然而止）
            - 以开放式选择列举结尾（"你怎么打、怎么选、放过谁、错信谁"）
            - 以隐含致命走向的悬念句结尾
      FAIL 触发：
        - 收尾是陈述句 + 句号
        - 收尾是总结性褒义句（"展现了武侠的魅力"）

输出：

  ✗ ⑤ 信息留白 缺失：第 3 段以陈述句收尾
```

```
对每张卡片：
  - 声明的 anchorRule 中的每一条原则编号，必须能在 value 文本中找到对应表达
  - 反之，value 中明显命中某条原则但 anchorRule 中未声明 → WARN

例：
  costume.anchorRule = [1, 3, 4]
  costume.value 中必须能找到：
    - 原则① 表达（如"X × Y"碰撞）
    - 原则③ 表达（如"vs""对比"剪影/材质句）
    - 原则④ 表达（如磨损/战损关键词）
  缺一即 FAIL
```

##### 5.11.3 三大快速人工测试（v2.4 新增 · 0.5 秒级硬卡点）

> Lance 二审环节，配合官网截图 + key art 进行实操级视觉测试。

```
┌──────────────────────────────────────────────────────────────────┐
│ 测试一 · 邮票测试（原则② 验证）                                    │
│   操作：把游戏 key art 缩到邮票大小（48x48px）                      │
│   PASS：仍能识别这是这款游戏 / 看出主色调与情绪色                     │
│   FAIL：糊成一团颜色，无识别度                                      │
├──────────────────────────────────────────────────────────────────┤
│ 测试二 · 剪影测试（原则③ 验证）                                    │
│   操作：把所有角色立绘涂成纯黑剪影（保留轮廓）                        │
│   PASS：能区分主角 / 敌人 / Boss / 阵营                             │
│   FAIL：所有剪影看起来差不多，区分不出来                             │
├──────────────────────────────────────────────────────────────────┤
│ 测试三 · 0.5 秒测试（原则①+② 综合）                                │
│   操作：把官网首屏 / Key Art 闪现 0.5 秒后立刻关闭                    │
│   问："你刚刚看到了什么？记住了什么？"                                │
│   PASS：能说出 1 个具体视觉记忆点（颜色/形象/奇观）                    │
│   FAIL：只能说"画面挺好看"或"想不起来"                               │
└──────────────────────────────────────────────────────────────────┘

任一测试 FAIL → aesthetic 模块需退回 Phase 4C 重写
```

##### 5.11.4 通过判定

```
原则①-⑤ 全部命中 + anchorRule 一致性 PASS + 三大测试全 PASS → PASS
任一原则缺失                                                  → 退回 4C 重写
任一测试 FAIL                                                 → 退回 4C 重写
WARN 项 ≤ 2                                                  → 进入人工二审
WARN 项 > 2                                                  → 退回 4C 调整
```

##### 5.11.5 美学综述 summary 审核（v2.5 新增 · 自动）

> 综述是整个 aesthetic 模块的统领段，必须先于五维卡片审核通过。

```
对 aesthetic.summary 三段 + references 数组校验：

  ├── ❌ label
  │   - 长度不在 [6, 20] 字 → FAIL
  │   - 缺失 → FAIL
  │
  ├── ❌ definition (Q1: 是什么美学)
  │   - 字数不在 [120, 180] → FAIL
  │   - 不包含至少 1 处"X × Y" / "X + Y" / "X 与 Y" 的碰撞句式（呼应原则①）→ FAIL
  │   - 不包含至少 2 个具体颜色词（呼应原则②）→ WARN
  │   - 出现"独一无二"/"绝无仅有"/"前所未有"等空洞自夸词 → FAIL
  │
  ├── ❌ evolution (Q2: 共识如何发展)
  │   - 字数不在 [150, 280] → FAIL
  │   - 不包含至少 2 个时间节点 / 阶段 → FAIL
  │     判定：出现至少 2 个"19xx" / "20xx" / "...年代"等时代标记
  │   - 不包含至少 3 部具体作品名（用《》或粗体标记）→ FAIL
  │   - 出现"由来已久"/"自古以来"/"源远流长"/"致敬经典" → FAIL
  │
  └── ❌ references (Q3: 借鉴谱系)
      - 数量不在 [5, 8] 条 → FAIL
      - 类型多样性 < 3 种 type → FAIL
      - 最早 year 与最晚 year 差距 < 20 年 → FAIL
      - 任一条 note 字数 < 15 或缺失 → FAIL
      - 任一条 note 仅描述作品本身、未指出"借鉴点" → WARN

输出：
  ✓ summary OK (label / Q1 / Q2 / Q3 全部通过)
  
或：
  ✗ evolution: 仅出现 1 个时间节点（"近年来..."），缺乏审美发展脉络
  ✗ references: 全是 game type，类型多样性不足
  → 退回 Phase 4C-1 重写
```

## 质量红线（不可妥协）

1. **截图必须来自开发商/发行商官网**，禁止使用 Steam/Epic/WeGame/Nintendo/PlayStation/Xbox 平台详情页
2. **每条体验观点必须有出处链接**，链接必须可达且内容对版
3. **B站视频必须逐条验证**：视频存在、内容与游戏完全契合、非蹭标签搬运
4. **信息源遵循"正反馈优先"原则**：权威媒体 > 高互动社区内容 > 补充来源
5. **内容加工以设计师视角为准**：术语专业、颗粒度到位、有审美判断
6. **WebP 格式统一**，不允许 PNG/JPG 直接入库
7. **审核通过后才能发布**，AI 验真 + Lance 二审双重兜底
8. **一款游戏一个独立 JSON 文件**，不混写
9. **竞品推荐零死链**，所有 rivals 中的游戏必须在数据库中有对应详情页
10. **所有文案必须有 `<b>` 高亮划重点**——playerExp 每条 text、story 每条 dims.value、aesthetic 每张卡片 value、rivals 每条 reason 至少包含 1 个 `<b>` 标签；**worldview 四维度 value 每条至少 2 个 `<b>`**；**story.overview 全篇累计至少 6 个 `<b>`**；加粗内容长度 ≤ 30 字符，禁止整句加粗
11. **worldview 必须通俗易懂**——四维度 `value` 字段拒绝书面化、拒绝产品文案口吻；必须像跟朋友讲游戏；Phase 5.7 黑话扫描 + 5.8 交付测试双重把关，FAIL 即退回重写
12. **story.overview 必须达到广告级转化标准**——核心口径：宣发文案不是内容摘要，而是商业广告，目的是 3 秒抓眼球 / 10 秒建立期待 / 30 秒促成转化；必须严格命中**五维钩子**（① 高概念碰撞 ② 极限处境 ③ 灰度悖论 ④ 情绪内核 ⑤ 信息留白），缺一不可；Phase 5.9 五维钩子逐项校验 + 5.10 三阶转化漏斗测试双重把关，FAIL 即退回 4B-1 重写
13. **aesthetic 必须达到 0.5 秒抓眼球标准 + 提供美学综述统领**——核心口径：美术风格如果不能在刷抖音的 0.5 秒内让玩家停下手指，宣发层面就已经输了；模块顶部必须有 `summary` 综述（label + definition + evolution + references[]），三问回答"是什么美学 / 共识如何发展 / 可借鉴哪些作品"；下方 5 维卡片必须严格命中**五大美术原则**（① 80/20 ② 色彩刺客 ③ 剪影定生死 ④ 信息磨损 ⑤ UI 即世界），每张卡片声明 anchorRule 且 value 内容能逐条对应；Phase 5.11 综述审核 + 原则命中校验 + 邮票/剪影/0.5 秒测试多重把关，FAIL 即退回 4C 重写
